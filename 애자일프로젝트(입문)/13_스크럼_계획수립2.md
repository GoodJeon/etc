# 스크럼 계획 수립2

## 작성(공부)일 : 2022-02-10(목)



### 유저 스토리(User Story)

* 개념
  * 사용자의 입장에서 정의한 기능에 대한 간단한 설명
  * 한 두 문장으로 구성되어있다.
  * 사용자가 이해할 수 있는 언어로 작성되어야 하며, 역할과 목표 정도만 표현
  * 작업량 산정의 용도로도 사용
* **핵심 구성요소**
  * 역할(Role) : Who?
  * 목표(Goal) : What?
  * 이유(Reason) : Why?

* 스토리 카드 : 대화의 증거, 공식문서 X
* 유저 스토리의 역할
  * 토론
  * 일정 관리
  * 산정
  * 기능 테스트
  * 완료 여부 판단
* 관리
  * 카드 혹은 포스트잇에 쓰여짐
  * **하나의 스프린트(반복, 주기, 반복)안에서 구현하고 테스트함**
  * 제품 책임자 혹은 고객이 작성
  * 자주 고객을 만나야함
  * **기능 관련 유저 스토리는 인수 테스트를 포함**
* 애자일 WBS(Work Bench Structure)
  * 테마(Theme) > 에픽(Epic) > 기능(Feature) > 유저 스토리(User Story)
  * 에픽(Epic) : 유저 스토리의 상위 개념
    * 유저 스토리와 관계된 **기능 또는 모듈의 집합**
  * 테마(Theme) : 에픽의 상위 개념으로 서브시스템, 제품 구성요소, 서비스 등을 말함

* What to do를 결정하는 제품 책임자(PO)

  * 제품 백로그로부터 해당 스프린트에 할당할 스프린트 백로그를 도출
  * 우선 순위에 따라 다음 스프린트에 작업할 백로그를 선정
  * 프로젝트 초기에는 PO가 기능 정의를 주도

* How to do를 결정하는 개발자/기능책임자(Feature Owner)

  * 각 기능을 정의, 개발, **스스로** 검증, 관리
  * 스프린트를 시작하면, 개발팀의 각 기능 책임자가 정의를 주도

* 효과적인 수집은..

  * 고객이 협력적이어야 함
  * 고객사 담당자가 대표성이 있어야 함
  * 고객사가 공식적으로 제공해야 함
  * 고객도 열정적으로 참여해야 함
  * 고객도 제품에 대한 이해를 해야 함

* 애자일 방법론

  | 애자일 방법론 | 제품의 정의 방법 | 학자                     |
  | ------------- | ---------------- | ------------------------ |
  | Scrum         | User Story       | 켄 슈와버, 제프 서덜렌드 |
  | XP            | USer Story       | 켄트 벡, 에릭 감마       |
  | Kanban        | Kanban Card      | 다이아치 오노            |
  | Crystal       | Use Case         | 엘리스테어 코번          |
  | FDD           | Feature Sets     | 피터 코드, 제프 드루카   |
  | ASD           | Feature Card     | 짐 하 이스미스           |

  

### 페르소나(Persona)

* 개념
  * 가상의 사용자 프로필
  *  유사한 특성이나 니즈를 가진 사용자 그룹의 표본을 보여주기 위해 가상의 개인 정보와 사진이 포함
  * 스크럼 팀은 사용자와 이해 관계자를 이해하기 위해 유저 스토리와 작업을 정의할 때 페르소나를 검토
  * 각 사용자의 역할에 맞는 사진을 추가해 동기부여하는 요소들을 기술
  * 개발 대상과 개발 방식의 구체화

* 정의 방법
  * 설문과 서베이, 박람회에서 인터뷰, 브레인스토밍 등
